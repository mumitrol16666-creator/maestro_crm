## 🔧 Бэкенд - Параллельные запросы к БД

### 1. POST /api/classes (создание занятия)
**Было** (последовательно):
```javascript
const group = await Group.findById()     // 300мс
const room = await Room.findById()       // 300мс  
const conflict = await Class.findOne()   // 400мс
const teacher = await Student.findById() // 200мс
// = 1200-1500мс
```

**Стало** (параллельно):
```javascript
const [group, room, conflict] = await Promise.all([
  Group.findById(),
  Room.findById(),
  Class.findOne()
]); // 400мс (время самого медленного)
const teacher = await Student.findById(); // 200мс
// = 600мс
```
**Ускорение: в 2.5 раза** 🚀

## 🎨 Применённые техники

### 1. Optimistic UI
**Принцип:** Показываем результат сразу, обновляем потом

**Примеры:**
- Закрываем модалку создания занятия → плашка появляется мгновенно
- Создаем заявку → новая строка добавляется сразу
- Конвертация → модалка результата показывается до получения ответа

**Обработка ошибок:**
- Откатываем изменения при ошибке
- Показываем уведомление
- Удаляем временные элементы

### 2. Promise.all - параллельные запросы
**Принцип:** Время = `max(запросов)`, а не `sum(запросов)`

**Фронтенд:**
```javascript
// Раньше:
const student = await fetch(...)  // 300мс
const stats = await fetch(...)    // 800мс
const membership = await fetch()  // 200мс
// = 1300мс

// Теперь:
const [student, stats, membership] = await Promise.all([
  fetch(...),
  fetch(...),
  fetch(...)
]); // = 800мс (самый долгий)
```

**Бэкенд:**
```javascript
const [group, room, conflict] = await Promise.all([
  Group.findById(),
  Room.findById(),
  Class.findOne()
]);
```

### 3. Ленивая загрузка (Lazy Loading)
**Принцип:** Показать UI → Загрузить данные в фоне

**Шаги:**
1. Открываем модалку мгновенно
2. Показываем "Загрузка..."
3. Загружаем данные асинхронно
4. Заполняем UI когда готово

### 4. Минимизация данных
- ✅ `.lean()` - plain JS объекты (быстрее)
- ✅ `.select()` - только нужные поля
- ✅ Убрали лишние `populate`
- ✅ Кэширование статичных данных (залы, направления)
- ✅ Кэширование статистики (2 минуты)

### 5. Умные обновления DOM
**Вместо полной перерисовки:**
- `table.insertBefore(newRow)` вместо `table.innerHTML = ...`
- `calendar.addEvent()` вместо `calendar.refetchEvents()`
- `setTimeout(..., 0)` для фоновых обновлений

## 💾 MongoDB на сервере - Рекомендации

### Текущий сервер:
```
CPU:  Xeon 1x2.60 GHz  ✅ OK
RAM:  1 GB             ⚠️ КРИТИЧНО МАЛО
Disk: 20GB SSD         ✅ OK
```

### Потребление с локальной MongoDB:
```
Node.js:   ~200-300 MB
MongoDB:   ~400-600 MB (минимум!)
Nginx:     ~50 MB
Система:   ~200 MB
─────────────
ИТОГО:     ~1000 MB (почти вся RAM!)
```

### ⚠️ Риски с 1GB RAM:
- **OOM Killer** убьет процесс при нехватке памяти
- **Swap на SSD** → очень медленно
- **Рост данных** → крэши
- **Нестабильность** системы

### ✅ Рекомендация: ATLAS M0 (текущее решение)

| Критерий | Atlas M0 | Локально (1GB RAM) |
|----------|----------|-------------------|
| **Стоимость** | $0/месяц | $0, но риски |
| **Память** | 512MB выделенная | Борьба за 1GB |
| **Надежность** | 99.9% uptime | Рискованно |
| **Бэкапы** | Автоматические | Нужно настраивать |
| **Мониторинг** | Встроенный | Нужно настраивать |
| **Скорость** | Оптимизированная | Ограничена RAM |
| **Масштабирование** | Легко | Нужно менять сервер |

### Если всё же нужно локально:
1. **Увеличить RAM до 2GB минимум** (лучше 4GB)
   - VPS с 2GB: ~$5-10/месяц
2. **Настроить MongoDB** с ограничением памяти:
   ```yaml
   storage:
     wiredTiger:
       engineConfig:
         cacheSizeGB: 0.3
   ```
3. **Добавить swap** на случай утечек

**Вывод:** Оставьте Atlas M0! 🎯
