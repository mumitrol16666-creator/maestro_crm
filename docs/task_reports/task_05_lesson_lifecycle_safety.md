# Задача: Устранение уязвимостей дублирования занятий, повторных списаний и возвратов в Maestro CRM

## 1. Исходный запрос (Промпт)
```
Нужно устранить уязвимости дублирования занятий, повторных списаний и повторных возвратов в жизненном цикле занятий Maestro CRM.

Проблема затрагивает создание занятий, генерацию расписания, подтверждение уроков, списания, переносы и отмены. Необходимо исправить уязвимости как на frontend, так и на backend.

1. Создание одиночных занятий
Сейчас при быстром двойном клике по кнопке создания занятия могут отправиться два параллельных запроса к POST /api/classes, из-за чего в базе появляются дубли одинаковых занятий.
Нужно исправить:
* заблокировать кнопку создания занятия сразу после первого клика;
* добавить флаг isSubmitting для формы создания занятия;
* повторный submit должен игнорироваться, пока первый запрос не завершён;
* кнопку можно разблокировать только после завершения запроса в finally;
* на backend добавить проверку на существование занятия с тем же временем, кабинетом, преподавателем и группой/учеником;
* при обнаружении дубля сервер должен возвращать ошибку и не создавать вторую запись.

2. Генерация занятий по регулярному расписанию
Сейчас при параллельном запуске генерации расписания двумя администраторами или в двух вкладках могут создаться дубли регулярных занятий.
Нужно исправить:
* защитить endpoint POST /api/classes/generate-from-schedule от параллельного выполнения одной и той же генерации;
* проверку существующих занятий и создание новых занятий выполнять атомарно;
* добавить защиту на уровне backend/БД, чтобы одинаковые занятия не могли быть созданы параллельными запросами;
* использовать транзакцию с подходящим уровнем изоляции или уникальные ограничения;
* повторный запуск генерации не должен создавать дубли уже существующих занятий.

3. Подтверждение урока и финансовое списание
Сейчас при одновременном подтверждении одного и того же урока двумя администраторами возможно двойное списание денег или двойное списание занятия с абонемента.
Нужно исправить endpoint POST /api/classes/:id/approve.
Требования:
* проверку текущего статуса занятия перенести внутрь транзакции;
* внутри транзакции заблокировать строку занятия на время обработки;
* если занятие уже завершено или уже было подтверждено, повторная транзакция должна остановиться без списаний;
* списания по балансу и абонементам должны быть идемпотентными;
* для одного урока и одного ученика нельзя создать два одинаковых списания;
* добавить уникальную защиту на уровне БД для транзакций списания;
* повторный запрос на подтверждение не должен повторно уменьшать accountBalance;
* повторный запрос на подтверждение не должен повторно уменьшать classesRemaining;
* статус занятия должен обновляться на completed только после успешного списания.

4. Перенос, отмена и возврат урока
Сейчас при двойном клике по действиям переноса, отмены или возврата могут отправиться параллельные запросы, из-за чего ученику может дважды вернуться занятие или сумма.
Нужно исправить:
* заблокировать кнопки жизненного цикла урока после первого клика;
* добавить флаг isSubmitting для действий переноса, отмены, возврата и повторного открытия;
* модальное окно подтверждения нужно закрывать до отправки запроса, чтобы исключить повторный клик;
* на backend выполнить возвраты внутри транзакции;
* внутри транзакции проверять текущий статус занятия;
* повторный перенос или отмена не должны повторно возвращать занятие на абонемент;
* повторный перенос или отмена не должны повторно возвращать деньги на баланс;
* возвраты должны быть идемпотентными.

5. Backend-защита от конкурентных запросов
Нужно добавить защиту не только на frontend, но и на backend, потому что frontend-блокировка не защищает от двух вкладок, двух администраторов или прямых API-запросов.
Требования:
* использовать транзакции для финансовых операций;
* выполнять проверку статуса занятия внутри транзакции;
* использовать блокировку строки занятия при критических операциях;
* использовать блокировку или атомарное обновление связанных абонементов и балансов;
* добавить уникальные ограничения для финансовых транзакций;
* повторная операция с тем же уроком и учеником не должна создавать вторую финансовую запись;
* при конфликте транзакции сервер должен возвращать корректную ошибку без частичных изменений.

6. Idempotency keys
Для критических операций нужно добавить ключи идемпотентности.
Применить для:
* создания занятия;
* генерации занятий;
* подтверждения урока;
* переноса урока;
* отмены урока;
* возврата урока;
* повторного открытия урока.
Требования:
* frontend должен передавать уникальный ключ операции;
* backend должен сохранять ключ операции на короткое время;
* повторный запрос с тем же ключом не должен выполнять операцию повторно;
* если операция уже выполнена, сервер должен вернуть результат без повторного изменения данных.

7. Уникальные ограничения в БД
Нужно добавить ограничения, которые не позволят создать дубли на уровне базы данных.
Проверить и добавить ограничения для:
* одинаковых занятий в одном кабинете на одно и то же время;
* одинаковых занятий у одного преподавателя на одно и то же время;
* одинаковых списаний по одному уроку и одному ученику;
* одинаковых возвратов по одному уроку и одному ученику;
* одинаковых membership-транзакций по одному уроку, ученику и типу операции.

8. Логирование и аудит
Для всех критических операций нужно добавить логирование.
Логировать:
* кто создал занятие;
* кто запустил генерацию расписания;
* кто подтвердил урок;
* кто отменил урок;
* кто перенёс урок;
* кто выполнил списание;
* кто выполнил возврат;
* была ли операция повторной;
* был ли запрос заблокирован из-за дубликата или конфликта.
```

## 2. Что было сделано

### База данных (БД) и ограничения
1. Добавлена новая модель `IdempotencyKey` в `prisma/schema.prisma` для кэширования ответов идемпотентных запросов (срок действия — 5 минут).
2. На уровне базы данных (PostgreSQL) в схему `schema.prisma` добавлены следующие составные уникальные индексы для исключения дублирования данных:
   - `Class`: `@@unique([roomId, date, startTime])` (одно занятие в комнате в одно время).
   - `Class`: `@@unique([teacherId, date, startTime])` (одно занятие у преподавателя в одно время).
   - `ClassAttendee`: `@@unique([classId, studentId])` (одна запись о присутствии студента на занятие).
   - `MembershipTransaction`: `@@unique([membershipId, classId, type])` (предотвращает повторные списания/возвраты по одному уроку, типу и абонементу).
3. База данных успешно обновлена при помощи команды `npx prisma db push`.

### Backend-защита и API
1. **Middleware идемпотентности (`src/middleware/idempotency.js`)**:
   - Реализована проверка и запись уникальных ключей операций (`X-Idempotency-Key`).
   - Если запрос с таким ключом уже обрабатывается, сервер возвращает статус `409 Conflict`.
   - Если операция уже завершена, сервер отдаёт сохраненный в БД кэшированный ответ без повторного выполнения логики.
   - middleware зарегистрировано на все критические эндпоинты в `classes.js`.
2. **Создание занятий (`POST /api/classes`)**:
   - Добавлены явные проверки занятости кабинета, преподавателя и группы.
   - Обработка исключений уникального индекса (`P2002`) возвращает корректную ошибку 400.
3. **Генерация расписания (`POST /api/classes/generate-from-schedule`)**:
   - Реализована защита от параллельного запуска генерации для одного и того же кабинета (проверяется список активных задач в памяти `generationJobs`).
   - Для фоновой генерации пакетами добавлена опция `skipDuplicates: true`, что позволяет безопасно пропускать дубликаты.
4. **Подтверждение урока (`POST /api/classes/:id/approve`)**:
   - Вся бизнес-логика (проверка статуса, списание баланса и абонементов) перенесена внутрь единой транзакции БД с блокировкой строки через сырой SQL-запрос `SELECT * FROM "Class" WHERE id = $1 FOR UPDATE`.
   - Если статус урока уже равен `completed`, транзакция прерывается без каких-либо изменений.
   - Изменение статуса на `completed` происходит только после выполнения всех списаний.
5. **Перенос, отмена, возврат и переоткрытие (`postpone`, `return-to-teacher`, `reopen`)**:
   - Все операции списания/возврата средств и абонементов завернуты в транзакции с блокировкой строк (`FOR UPDATE`).
   - Добавлены проверки текущего статуса урока внутри транзакций, предотвращая повторные списания или начисления средств.

### Frontend UI и взаимодействия
В файле `frontend/js/modules/schedule/schedule.js` добавлены следующие изменения:
1. **Защита кнопок и форм**:
   - Добавлены глобальные флаги отправки: `isClassSubmitting`, `isApprovingClass`, `isLifecycleSubmitting`.
   - Кнопки форм создания, генерации и действий над уроком отключаются сразу после клика и разблокируются только в блоках `finally`.
2. **Быстрое закрытие модальных окон**:
   - Модальное окно деталей/посещаемости (`closeAttendanceModal()`) и окно деталей расписания (`closeScheduleDetails()`) закрываются моментально до выполнения сетевого запроса, предотвращая повторные клики.
3. **Ключи идемпотентности**:
   - На каждый запрос генерируется новый UUID-like ключ и передается в заголовке `X-Idempotency-Key` для создания, генерации, подтверждения, отмены, переноса и возврата занятий.
4. **Сброс кэша браузера**:
   - В `admin.html` версия скрипта обновлена до `schedule.js?v=137`.

### Логирование и аудит
1. В `idempotency.js` добавлены логи в `ActivityLog` для событий кэш-хитов (`idempotency_cache_hit`) и параллельных блокировок (`idempotency_conflict_in_progress`, `idempotency_conflict_parallel`).
2. В `classes.js` добавлены логи в `ActivityLog` при создании занятий, старте генерации расписания, успешном завершении генерации, возникновении конфликтов расписания.
3. Логирование действий пользователей при переносах, отменах, возвратах и переоткрытиях выполняется внутри соответствующих транзакций.

## 3. Список измененных файлов
- `backend/prisma/schema.prisma` — схемы и уникальные индексы БД.
- `backend/src/middleware/idempotency.js` — middleware идемпотентности и логирование повторных запросов.
- `backend/src/routes/classes.js` — роуты создания, генерации, подтверждения и переноса с атомарными транзакциями, row-level locking и логированием.
- `backend/src/services/lessonLifecycle.js` — транзакции с row-level locking для повторного открытия и возврата уроков.
- `frontend/js/modules/schedule/schedule.js` — блокировка интерфейса, закрытие модалок перед запросом, генерация `X-Idempotency-Key`.
- `frontend/public/admin.html` — инкремент версии скрипта для сброса кэша.
